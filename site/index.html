Get started

Mount the first workspace.

$ curl -fsSL https://drive9.ai/install.sh | sh
$ drive9 create
$ drive9 fs mkdir :/skills
$ mkdir -p ~/drive9-skills
$ drive9 mount :/skills ~/drive9-skills

Core use case

Claude Code writes. Codex reads. No sync glue.

Before

Files stay trapped inside one tool.

Skills, artifacts, and working state land in tool-specific local folders. Sharing means copy scripts, env vars, or manual handoff.

After

Agents share one remote namespace.

Mount :/skills into each agent environment. One agent writes a file; the next agent reads the same path.

Built on top

Search and organize, built on the agent filesystem.

First, agents share files. Then supported files can be indexed so people, apps, and agents search what the workspace produces.

Team knowledge workspace

Put docs and runbooks where agents can mount them. Search supported content with drive9 fs grep.

Searchable asset library

Use the same filesystem as storage substrate; add search over supported files when the product needs it.

Details

Keep the model simple.

Is drive9 a cloud drive?

It is a filesystem layer for agents first. Human-facing search and knowledge workflows are built on that layer.

How is it different from S3?

S3 is blob storage. drive9 gives agents directories, paths, mount, and file operations.

How is search handled?

Search is an upper-layer workflow over supported files in the workspace, not the first thing agents need to understand.
