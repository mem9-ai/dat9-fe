# drive9

You have a persistent workspace.

  • Use it when files must survive this session.
  • Use it when another agent needs your artifacts.
  • Use it when the user asks to find saved work.

The workspace is addressed with shared paths.

For agents, drive9 starts as a skill file, not a setup screen.

Agent interface

The skill is the agent-facing interface.

Agents read the skill once and know when to use drive9, where shared files live, and how to pass artifacts to the next agent.

01

Read the skill

The agent learns the drive9 contract: workspace paths, persistence rules, and handoff behavior.

02

Use the workspace

The agent stores files, reads prior work, and searches saved content through the skill instructions.

03

Hand off files

Another agent with the same skill can continue from the same workspace.

Core use case

Claude Code learns the skill. Codex uses the same workspace.

Before

Each agent keeps files in its own context.

Skills, artifacts, and working state stay trapped in tool-specific local folders or session memory.

After

Agents share one filesystem skill.

The skill tells agents how to use the workspace. One agent writes an artifact; the next agent reads it from the same path.

Setup once

A human wires the filesystem in.

CLI setup prepares the workspace that the skill refers to. It is not the agent-facing product story.

$ curl -fsSL https://drive9.ai/install.sh | sh
$ drive9 create
$ drive9 fs mkdir :/skills
$ mkdir -p ~/drive9-skills
$ drive9 mount :/skills ~/drive9-skills

Details

Keep the model simple.

Is drive9 a cloud drive?

It is a filesystem skill and workspace for agents first. Human-facing search and knowledge workflows are built on that layer.

How is it different from S3?

S3 is blob storage. drive9 gives agents directories, paths, mount, and file operations.

How is search handled?

Search is an upper-layer workflow over supported files in the workspace. The first thing agents need is the filesystem skill.
